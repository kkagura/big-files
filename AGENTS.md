# 智能体开发约定

本文件适用于整个项目。任何参与本项目开发的智能体都必须遵守以下规则。

## 1. 代码与设计文档保持同步

- 开始修改代码前，先阅读与该功能相关的设计文档，当前主设计文档为 `AI_ANALYSIS_DESIGN.md`。
- 如果代码实现、接口、目录结构、配置格式、交互流程、安全边界或其他行为与设计文档不一致，必须在同一次变更中同步更新设计文档。
- 不允许在明知实现已经偏离设计文档的情况下只修改代码。
- 如果实现过程中需要改变原设计，应先确认新的设计表达清楚，再按照更新后的设计实现。
- 仅修正文案、注释或不影响设计的内部实现细节时，不要求机械修改设计文档；但必须确认不存在行为或架构变化。

## 2. 功能与测试保持同步

- 新增或修改任何功能时，必须新增或更新对应的测试文件。
- 测试至少覆盖本次变更的主要正常流程；涉及错误处理、输入校验、安全边界或回归风险时，也必须覆盖相应场景。
- 修复缺陷时，应优先增加能够复现该缺陷的回归测试，再完成修复。
- 不允许仅因为当前项目缺少测试框架或已有测试不足，就跳过本次功能对应的测试；应补齐必要的测试基础设施。
- 纯文档修改、纯注释修改以及不改变程序行为的格式化，可以不新增测试，但仍需执行适用的现有检查。

## 3. 完成功能前必须运行测试

- 完成某项功能开发后，必须先运行与本次变更直接相关的测试。
- 在条件允许时，还必须运行项目完整测试套件。当前 Go 项目的完整测试命令为：

  ```powershell
  go test ./...
  ```

- 如果项目后续增加静态检查、集成测试或端到端测试，应同时执行与变更相关的检查命令。
- 只有相关测试通过后，才能将功能声明为完成。
- 如果测试因环境、依赖或外部服务限制无法运行，必须明确说明未运行的测试、具体原因和潜在风险，不能将“未运行”表述为“已通过”。
- 如果测试失败，必须说明失败项；不得隐瞒失败结果或在未解决失败时声称功能已经完成。

## 4. 交付说明

每次功能开发完成后的说明至少应包含：

- 实现或修改了什么。
- 同步更新了哪些设计内容。
- 新增或更新了哪些测试。
- 实际执行了哪些测试命令及其结果。
